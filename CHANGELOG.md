## [Unreleased]

### Fixed

- **Ask sessions no longer fail with a misleading "embedding not configured" message after 3–5 turns** — The handler in `services/api/internal/handler/search.go::Ask` previously hard-trimmed conversation history to the last 10 turns with no awareness of message length or model context window, so by turn 5 of a normal session the upstream provider could 4xx with `invalid_request_error: prompt is too long`. Every non-2xx from the provider was then mapped to a flat HTTP 500 with the generic error `"LLM synthesis failed"`, which the dashboard rendered as `"Sorry, I could not answer this question. Make sure embedding is configured for this project."` — a copy-paste that suggested the wrong root cause (embedding, not context). The handler now sizes every Ask call against the model's published `MaxInputTokens`, subtracts `max_tokens` (the 2048-token generation reserve) and ~600 tokens for the system prompt, then trims session history newest-first by token count rather than turn count. Insights are dropped lowest-scoring-first if the retrieved context alone wouldn't fit. The trim invariants are: never separate a user message from its assistant reply (no orphan assistants), the current user question always rides at the end, and the safety margin widens from 5% to 15% when the active counter is approximate so under-counting can't push us past the upstream cap.

- **Agent K8s Job pods satisfy PodSecurity `restricted` admission** — `KubernetesRunner.buildJob()` now stamps a full pod + container `SecurityContext` (`RunAsNonRoot=true`, `RunAsUser/Group=1000`, `FSGroup=1000`, `AllowPrivilegeEscalation=false`, `ReadOnlyRootFilesystem=true`, `Capabilities.Drop=[ALL]`, `SeccompProfile.Type=RuntimeDefault`) on every Job it creates (discovery runs, schema indexing, test-connection). Without these fields, namespaces labeled `pod-security.kubernetes.io/enforce=restricted` reject every agent Job and the dashboard shows "Schema index: Not indexed" indefinitely. To keep `ReadOnlyRootFilesystem=true` compatible with cloud SDKs that opportunistically write disk caches (gosnowflake OCSP, AWS/GCP token files), the spec also declares an `emptyDir` volume mounted at `/tmp` and points `TMPDIR=/tmp` + `HOME=/tmp`. The agent image (Alpine + static Go binary) already runs as UID 1000 with no capabilities — this is a spec-only fix with no runtime behavior change.

### Added

- **`MaxInputTokens` + `Encoding` fields on every LLM model catalog entry** — Every shipped model in `libs/go-common/llm/registry.go::ModelEntry` now carries an upstream-published context-window size and (for OpenAI-wire models) a tiktoken encoding name. Resolution is catalog → ProviderMeta `DefaultMaxInputTokens` → package-level `DefaultMaxInputTokens = 32000` so unknown models still get a usable (conservative) budget. Claude 4.x is 200K, GPT-5 family is 400K, GPT-4.1 family is 1M, GPT-4o is 128K, o-series is 200K, Gemini 1.5/2.x is 1M, and Bedrock OSS models declare per-family caps. Exposed on `/api/v1/providers/llm` so the dashboard can render the window in the model picker if it wants. `GetMaxInputTokens(provider, model)` and `GetEncoding(provider, model)` are the package-level lookup helpers.

- **`gollm.TokenCounter` interface + per-provider implementations** — A new optional capability interface, `gollm.TokenCounterProvider`, lets a Provider hand back an exact counter for a given model. Exact counters are wired for:
  - **Claude direct** via `/v1/messages/count_tokens` (one extra RTT, exact for the active model).
  - **OpenAI direct** via `gollm.NewTiktokenCounter` (factored out into `libs/go-common/llm/tiktoken_counter.go` so any OpenAI-wire provider can reuse it). Routes to `ApproximateCounter` when `base_url` points at a non-canonical host (self-hosted proxy / OpenAI-compatible gateway) — tiktoken can't be claimed exact for an unknown upstream tokenizer.
  - **Azure AI Foundry** via the same shared `NewTiktokenCounter` for OpenAI-wire models (GPT-5 / 4.1 / 4o family). Anthropic-wire entries on Foundry and unknown deployment names fall back to `ApproximateCounter` — Foundry doesn't expose Claude's count_tokens through its surface.
  - **Vertex AI Gemini** via the `:countTokens` REST endpoint on `publishers/google/models/<model>`. One extra RTT, exact for the active model, uses the same ADC bearer the Chat path uses. Claude-on-Vertex and the OpenAI-compat MaaS models (Llama / Qwen / DeepSeek / Mistral) fall back to `ApproximateCounter` — Vertex's Anthropic publisher doesn't expose a public count_tokens, and the SentencePiece-based MaaS models would be systematically mis-counted by tiktoken.
  - **Bedrock, Ollama** continue to use `gollm.ApproximateCounter` (rune count / 4) — neither exposes a universal count API. The budget layer widens the safety margin from 5% to 15% to absorb the inaccuracy.
  The interface is optional precisely so a new provider can ship without a counter and still benefit from the budget walk on day one.

- **`docs/concepts/ask.md`** — New page documenting the budget algorithm, per-provider counter choices, the typed error codes (`embedding_not_configured`, `llm_not_configured`, `context_overflow`, `llm_upstream`, `llm_synthesis_failed`), and what users see for each. Cross-linked from `docs/reference/configuration.md`.

### Changed

- **`POST /api/v1/projects/{id}/ask` returns typed error codes** — Previously every failure was a flat HTTP 500 with `error: "LLM synthesis failed"`. The handler now returns a `code` field on the response body alongside the human-readable `error`, plus an optional `details` field carrying provider-sanitised context. Status codes: `412 embedding_not_configured`, `412 llm_not_configured`, `413 context_overflow` (the case the budget walk surfaces when even minimal context won't fit), `502 llm_upstream` for rate-limit / quota / content-filter, `504` for cancel / deadline, `500 llm_synthesis_failed` for unclassified provider failures. Existing callers that only look at the human `error` keep working — the new `code` field is additive.

- **Dashboard `ApiError` carries `code` + `details`** — `ui/dashboard/src/lib/api.ts` now throws a typed `ApiError` (extends `Error`) on every non-2xx response with the parsed `code` and `details` fields attached. The Ask page branches on `code` to surface specific copy: configuration mismatches point at project settings, context-overflow points at "start a new chat or pick a wider model", and upstream errors show the sanitised provider message. The generic "could not answer" copy is now only the fallback for unknown codes or non-`ApiError` throwables.

### Tests

- **Real-provider Ask integration coverage across Bedrock, OpenAI, Vertex AI Gemini, and Ollama** — `services/api/internal/handler/ask_real_providers_integration_test.go` (renamed from `ask_bedrock_integration_test.go`) now runs the full Ask flow — budget walk, RAG fitting, history trim, typed-error machinery — against four real LLM backends via a MongoDB testcontainer. Per provider: a happy-path "new session" test and a 50-turn "trims long session" test. Each suite skips cleanly when its credentials env var is absent (`INTEGRATION_TEST_BEDROCK_REGION`, `INTEGRATION_TEST_OPENAI_API_KEY`, `INTEGRATION_TEST_VERTEX_PROJECT_ID`, `INTEGRATION_TEST_OLLAMA_HOST`), so non-cloud CI runs them as skips. Shared helpers (`askMockVectorStore`, `askMockEmbedding`, `askSearchHandlerForProvider`, `runAskHappyPath`, `runAskTrim`) keep per-provider sections small. Verified passing locally against Claude Haiku 4.5 (Bedrock), gpt-4o-mini (OpenAI), Gemini 2.5 Flash (Vertex), and qwen3:32b (self-hosted Ollama).

- **tiktoken vs real OpenAI billing cross-check** — `providers/llm/openai/tokens_integration_test.go` sends a ~100-token prompt to OpenAI, reads `prompt_tokens` from the response, and asserts the local tiktoken count is within ±20% of OpenAI's billing. Catches a counter regression where the wrong encoding table would silently drift. Plus a monotonicity test that progressively longer prompts produce strictly larger counts. Observed drift on gpt-4o-mini: ~7% (well inside the safety margin).

## [0.5.0] - 2026-05-15

### Added

- **Platform-enforced claim-discipline rules appended to every writer and verifier prompt at runtime** — A new `services/agent/internal/discipline` package owns four exported rule blocks (base context, analysis, recommendations, verifier) that the orchestrator and the insight validator append to their respective prompts immediately before sending to the LLM. The rules cover scope-binding superlatives, no-claim-beyond-query-window, re-ranking from raw result rows, addressing counter-evidence explicitly, cross-field self-consistency, citing the step for every number, partial-period hygiene, and objective non-dramatic prose (principle + worked Bad → Good examples, no enumerated banned-words list). The verifier block additionally carries V1–V4 rules that funnel through the existing count-comparison pipeline: V1 instructs the LLM to construct ranking SQL whose returned `count` encodes claim correctness (`SELECT CASE WHEN actual_top = claimed_top THEN <claimed> ELSE 0 END AS count`); V2/V3/V4 emit `SELECT 0 AS count` with an in-SQL comment naming the offense, so the existing `count == 0 → rejected` branch fires with a logged reason. **Why in code, not in pack files:** user-added custom analysis areas (created via the dashboard or `PUT /api/v1/projects/{id}/prompts`) carry no pack content; user-edited or generator-produced packs can drop the rules by accident; cross-repo drift between filesystem packs and the enterprise pack-generator meta-prompts is eliminated. The rule text is intentionally language-neutral — every rule block carries an explicit "LANGUAGE NOTE" telling the LLM to apply the same constraints to the project's configured output language (the English examples are illustrations of the principle, not the principle). The three orchestrator injection sites in `services/agent/internal/discovery/orchestrator.go` (base context after substitution, each analysis-area prompt after assembly, recommendations prompt after assembly) call `discipline.AppendBaseContextRules` / `AppendAnalysisRules` / `AppendRecommendationsRules`; the verifier wires `discipline.VerifierRules()` into `buildVerificationPrompt`'s `fmt.Sprintf`. No model schema change, no template-variable change, no backfill migration — code-level injection makes every existing project benefit on its next run regardless of stored prompt content. Coverage: 18 unit tests in `discipline/rules_test.go` (rule-text shape, leading markdown header, language-neutrality, output-language clause, deterministic returns, no leftover template variables, valid UTF-8, non-overlapping section headers, Append helpers' double-newline separator + empty-input + double-call behavior + custom-area regression with empty / multi-line / unsubstituted / non-Latin payloads), plus 5 validator tests in `insight_validator_discipline_test.go` covering single-shot mode, loop mode, lookup-notice path, preservation of the existing prompt contract (datasets header, dialect header, `AS count` alias, `COUNT(DISTINCT user_id)`, fully-qualified table refs), and a regression test fixture for the case-study insight that motivated the work. Documentation: `docs/guides/customizing-prompts.md` gains a new "Platform-Enforced Claim Discipline Rules" section explaining what the platform appends and that the dashboard prompt editor is NOT the full prompt the LLM receives.

- **`ExecutiveSummary` model + shared `<CitationsFooter>` dashboard component** — A new structured-document type in `libs/go-common/models/exec_summary.go` lays out the newspaper-style discovery summary the upcoming enterprise plugin will produce (issue masthead, lead article + 3-up stories, themed section pages with bar lists and stat rows, 30/90-day action plan, "what we did" closer, and `Citation` back-references on every quantitative claim). The model carries no behaviour — the community build defines the type so the dashboard can render an enterprise-supplied document without an overlay-time type-shim. JSON tag and `omitempty` discipline are pinned by 11 unit tests covering full-fixture round-trip, every nested section, near-empty-document compactness, and the citation shape contract. A new `ui/dashboard/src/components/citations/CitationsFooter.tsx` component pairs with it: the same numbered "Sources" footer the Ask reply card used to render inline, factored into a shared component so both the Ask card and the future newspaper page resolve citation deep-links through `sourceHref()` and number them in first-seen, deduplicated order via the exported `numberSources()` helper. The Ask page is refactored to consume the new component (no behaviour change there). Tests: 20 Jest tests covering `sourceHref` URL shape (insight vs recommendation, discovery_id present vs missing), `numberSources` dedupe and (type,id) separation, footer rendering (empty input renders null, default and custom heading, sequential numbering, dedupe collapse, deep links, score toggle, name/title fallback, severity badge presence gating, both source types, caller-controlled ordering).

- **Discovery run completion plugin hook** — A new fifth plugin hook (`apiserver.RegisterRunCompletionHook`) fires once per discovery run that reaches a terminal state (`completed`, `failed`, or `cancelled`). Plugins register a named hook from `init()`; the API spins up a 15-second-tick background dispatcher (only when at least one hook is registered — zero hooks means zero Mongo scans) that scans `discovery_runs` for terminal rows whose new `completion_hooks_fired_at` field is unset, fires every registered hook in registration order, and stamps the field once every hook returns nil. Hooks are sequential within a run (an audit-record hook finishes before a downstream notifier observes it) and resilient (a hook that panics is recovered with a panic-tagged error; sibling hooks still run). Partial-failure semantics: a non-nil return from any hook leaves the run unmarked so every hook re-fires on the next tick — hooks are documented as idempotent precisely so a peer failure doesn't double-trigger the peers that already succeeded. The payload (`apiserver.RunCompletion`) carries `RunID`, `ProjectID`, `Status`, `CompletedAt`, and `Error` (set only for failed runs); insights and recommendations are persisted before the agent flips the run to `completed`, so consumers may rely on those collections being readable when the hook fires. A new compound index `(status, completion_hooks_fired_at, started_at)` on `discovery_runs` keeps the per-tick dispatcher query bounded even with tens of thousands of historical rows. The agent-side `DiscoveryRun` model mirrors the field so the shared collection schema stays consistent. Documented in `docs/concepts/plugin-hooks.md` as Hook 5. Coverage: registry unit tests (panic on nil/dup/empty-name, ordered fire, error capture, panic recovery, concurrent register/fire), apiserver wrapper tests (delegation, panics, reset), dispatcher tests (no-runs / list-error / payload / nil-CompletedAt / all-succeed / any-fail / partial-failure / hook-panic / mark-error / cancel-on-context), and Mongo testcontainer integration tests on the two new repo methods (filter shape, FIFO order, limit, default-limit, no-matches, mark sets field, mark removes from list, invalid run ID errors, mark touches updated_at).

- **Cached-schema filter plugin hook + run-time filter integration in the orchestrator** — `agentplugin.RegisterCachedSchemaFilter` and `ApplyCachedSchemaFilters` are a fourth plugin hook (alongside the three from the prior release) that fires every discovery run, after the orchestrator loads schemas from `project_schema_cache` and before the catalog goes to the LLM. Different trigger point from `ListTablesFilter`: that one runs during a fresh schema-index pass (constraining what gets cached); this one runs every run on already-cached schemas (constraining what the LLM sees). Together they let a downstream allow-/deny-list (e.g. discovery scope) take effect for both already-indexed and freshly-indexed projects without re-indexing. The orchestrator sorts the input keys ascending before calling filters (deterministic across runs despite Go's randomised map iteration), validates the output is a subset of the input (a misbehaving filter that returns a key not in the cache is rejected with a clear error rather than surfacing a phantom table to the LLM), and surfaces a "review the discovery scope" error when the filter chain drops every table rather than letting an empty catalog hit the LLM. Also clips `CacheSchemaProvider.Search` hits to the (already-filtered) schemas map so `search_tables` cannot surface tables that `lookup_schema` would then report as Not Found. New documentation in `docs/concepts/plugin-hooks.md` (Hook 4 section). Tests: in-package unit tests for the new registry mirroring the ListTables filter coverage (no-filter passthrough, nil/empty/duplicate panics, chain ordering, error abort, projectID plumbed); orchestrator unit tests covering filter shrinks catalog, filter error surfaces, drop-all returns the explicit guidance error, and invented-key rejection.

- **`GET /api/v1/projects/{id}/schema-cache/tables` endpoint** — exposes the distinct `schema_key` values the agent has cached for a project (one entry per qualified table — `<dataset>.<table>` for BigQuery, `<schema>.<table>` for Postgres / Snowflake / Databricks, `dbo.orders` for MSSQL — i.e. whatever the warehouse provider chose to canonicalise on). Wraps the `{"tables": [...]}` payload in the API's standard `data` envelope, so the wire shape is `{"data": {"tables": [...]}}`. Tables are sorted ascending; an empty cache renders as `[]` (not `null`) so the dashboard's mapping helpers don't need to special-case nil. Backed by a new `Repository.ListTables` method on the existing `project_schema_cache` collection — read-only; the agent owns writes. Used by the discovery-scope page's table picker so the user picks from what the agent actually sees, not a free-form text input. Tests: handler unit tests (happy / empty / no-repo / missing-project / empty-project-id / repo-error) plus testcontainer integration tests (distinct-and-sorted across multiple `warehouse_hash` generations, empty-cache returns `[]`, isolated-across-projects, empty-projectID rejected).

- **Three generic plugin hooks for downstream agents** — A new `libs/go-common/agentplugin` package exposes `RegisterContextProvider` / `GetAllContextProviders` so plugins can append markdown sections to agent prompts in a deterministic order, with per-provider error containment and panic recovery. The existing knowledge-sources retriever (`libs/go-common/sources`) registers itself through this hook in `init()` so the orchestrator's prompt assembly walks the registry instead of hardcoding a single retriever; a regression test guards byte-for-byte equivalence of the rendered output. A second hook, `discovery.RegisterListTablesFilter`, runs after the warehouse driver lists per-dataset tables and before per-table schema discovery — plugins can shrink the set (allow-/deny-lists, regex filters) without forking `schema_discovery.go`. A third hook, `apiserver.RegisterAskOverride`, lets a plugin replace the community handler for `POST /api/v1/projects/{id}/ask` so agentic / tool-using flows can layer on without touching the community route. All three names are intentionally feature-agnostic and additive: with no plugin loaded, the registries are no-ops or return empty output (the community knowledge-sources provider registers itself as a default at init() and emits an empty section when no sources are configured) and behavior is unchanged. Documented in `docs/concepts/plugin-hooks.md`.

- **`### Key Learnings` prompt header renamed to `### Agent observations`** — The agent's auto-learnings (`ProjectContext.Notes`) used to render under "Key Learnings" inside `## Previous Discovery Context`. The new label disambiguates them from user-authored knowledge sources, which render under `## Project Knowledge`. Field names and storage are unchanged; only the prompt-section header was renamed (`services/agent/internal/discovery/orchestrator.go`).

- **Three new feature-flag wire constants in `libs/go-common/policy`** — `policy.FeatureSlack` (`slack_enabled`), `policy.FeatureSources` (`sources_enabled`), and `policy.FeaturePackGen` (`pack_gen_enabled`) join the canonical `AllFeatures` slice. Self-hosted OSS deployments are unaffected (the default `NoopChecker.FeatureEnabled` continues to return `true` regardless of the flag), but downstream `Checker` implementations — both the cloud control plane and the upcoming enterprise license plugin — can now express entitlement for Slack notifications, file-upload knowledge sources, and LLM-generated domain packs through the same single source of truth as audit, governance, and customer-IdP SSO. Pure additive change: no behavior change for any existing caller, no interface changes, no env var changes.

- **`GET /api/v1/me`** — Universal endpoint returning the authenticated `UserPrincipal` (sub / email / org_id / roles) read from the request context. Single source of truth for "who am I" surfaces in any dashboard: NoAuth deployments get the anonymous principal, OIDC deployments get the customer-IdP user, and cloud-enterprise tenants get whichever validator (cloud-SSO chain or OIDC) just authenticated the request — the handler is auth-backend-agnostic. Returns 401 when no principal is in the context (defensive for misconfigured chains). Gated on the `viewer` role so any authenticated user can ask.

- **`auth.PeekProvider()` for chain-aware plugins** — `libs/go-common/auth` gains a `PeekProvider()` accessor that returns the currently registered provider as-is (nil if none). Unlike `GetProvider()` it does not fall back to `NoAuthProvider`, so plugins that want to chain on top of an existing provider — e.g., the cloud-auth plugin in the cloud-enterprise tenant image where it must coexist with enterprise OIDC — can detect whether to wrap or replace. Self-hosted deployments and the existing single-provider plugin pattern are unaffected.

- **Per-project output language for insights, recommendations, and `/ask`** — Projects gain a `language` field (default `English` when empty) that controls the human-readable output language for narrative fields (insight names/descriptions, recommendation titles/descriptions, summaries, `/ask` answers) without affecting analysis quality. The agent's `base_context` prompt grows a `{{LANGUAGE}}` placeholder substituted alongside `{{PROFILE}}` and `{{PREVIOUS_CONTEXT}}`; the `/ask` system prompt at `services/api/internal/handler/search.go` is rendered with `Project.EffectiveLanguage()` so synthesized answers match the configured language. Technical tokens (SQL, column names, identifiers, JSON keys, severity values, analysis_area IDs, citation markers) stay in English regardless of the choice — the directive is explicit in every domain-pack `base_context.md` so the LLM translates context that may already be in another language rather than mirroring it. Settings → General gets a searchable Select with 20 languages (English, Türkçe, Deutsch, Español, Français, Italiano, Português, Nederlands, Polski, Русский, العربية, עברית, 日本語, 한국어, 简体中文, 繁體中文, हिन्दी, Bahasa Indonesia, Tiếng Việt, ไทย). All four bundled domain packs (ecommerce, gaming, social, system-test) are reseeded with the new directive; the pack generator (enterprise) requires generated packs to include `{{LANGUAGE}}` in `base_context` so newly generated packs ship language-aware. Legacy projects with empty `Language` continue to behave exactly as before — `EffectiveLanguage()` falls back to `English`, and existing custom packs without `{{LANGUAGE}}` keep working (the substitution is a no-op).

- **Domain pack generation hooks** — A new `libs/go-common/packgen` package introduces a `Provider` interface, registry, no-op default, and shared request/result types for LLM-driven domain pack generation. The agent gains a `--mode pack-gen` flag that loads the configured Provider and runs `Generate(...)` end-to-end for the project; the API gains `POST /api/v1/projects/{id}/pack-generate` (kicks off generation) and `POST /api/v1/projects/{id}/pack-generate/regenerate` (synchronous section-level regeneration). Both endpoints return 404 when no Provider is configured. Project documents grow a `state` field (`pack_generation_pending` → `pack_generation` → `pack_generation_done` → `ready`), a `generate_pack` payload carrying the user's pack name/slug/description, and a `pack_gen_last_error` field that records the most recent failure (3-retry-exceeded validator error or LLM error) so the wizard can surface it for retry — the orchestrator reverts state to `pack_generation_pending` and writes the error message on any failure, and clears the field on the next successful generate. Legacy projects with empty `state` are treated as `ready` so existing deployments keep working without a backfill migration. `POST /api/v1/projects` accepts `generate_pack: { enabled: true, pack_name, pack_slug, description? }` and skips domain-pack lookup, prompt seeding, and schema-index enqueue for wizard projects — those steps run later when generation completes. The package ships with 100% statement coverage and a contract that lets a plugin register a real generator via `init()` + blank import without touching call sites. The dashboard adds a parallel **"Generate one for me"** mode on `/projects/new`, a three-step wizard at `/projects/{id}/generate` (knowledge sources, warehouse + providers, launch), and a state-aware project page that hides discovery while a project is in any pack-generation state and renders a draft-pack preview with per-section regenerate textareas once the agent finishes. The settings page is split behind reusable `<WarehouseConfigPanel>` and `<ProvidersPanel>` components shared between the standalone settings page and the wizard, and those panels now delegate their form rendering to `<WarehouseFormFields>` and `<LLMFormFields>` — controlled components also consumed directly by the new-project wizard at `/projects/new` so the three entry points (new-project wizard, settings tab, pack-gen wizard) share a single source of truth for provider/auth/credential rendering and metadata-driven defaults (MSSQL renders "Schema" with `dbo` / `encrypt: true` / `trust_server_certificate: false` pre-filled, BigQuery renders "Datasets" with the comma-separated hint, etc.). The draft-pack preview also renders a compact "what's unique vs the nearest built-in pack" diff summary (Jaccard category-name similarity picks the baseline; new categories, analysis areas, and profile fields are listed inline), each regenerate textarea now ships with a section-specific feedback hint and a "regenerated" badge that surfaces the last feedback used for that section, and failed-attempt errors persist on the project so the wizard's confirm step shows them inline with retry guidance. New docs page `docs/guides/generating-domain-packs.md` walks through the user-facing flow; `docs/concepts/discovery-lifecycle.md` documents the project state machine; `docs/concepts/architecture.md` and `docs/reference/configuration.md` document the new agent run mode; `docs/reference/api.md` documents the new endpoints.

- **On-demand schema retrieval in the exploration agent** — The agent no longer dumps a Level-1 block (per-table column lists + sample rows for the top-K retriever matches) into the system prompt of every step. Two new actions let the model pull L1 detail only for tables it actually wants to use: `lookup_schema` (up to 10 fully-qualified refs per call, served from the in-memory schemas cache — no warehouse traffic) and `search_tables` (semantic Qdrant query when the catalog hint isn't enough). Per-run budgets: 30 lookups, 30 searches; topK clamped to 30. Replaces the previous architecture that exhausted the Bedrock 1M-token context at ~step 98 on wide-warehouse runs (`prompt is too long: 1002763 tokens > 1000000 maximum`). Every domain-pack exploration prompt (gaming, social, ecommerce, system-test) was rewritten to teach the new action contract. Telemetry: per-action counters `schema_lookup_calls` and `schema_search_calls` on `discovery_runs` replace the old single `schema_inspect_table_calls`. `{{SCHEMA_INFO}}` is now the canonical (and only) schema variable; it renders the Level-0 catalog only. The `{{SCHEMA_CATALOG}}` and `{{SCHEMA_RETRIEVED}}` placeholders introduced earlier in this release cycle are gone. New `docs/architecture/agent-on-demand-schema.md` documents the rationale, flow, file map, telemetry, and tests. Existing customised prompts on existing projects must be re-saved from the new defaults to pick up the action contract.

- **Schema indexing + Qdrant retrieval for /ask** — Projects build a persistent, searchable schema index at creation time (blurbs + embeddings in Qdrant). The index serves both discovery (catalog + on-demand `search_tables`) and `/ask` (top-K retrieval per question). Project creation grows an "indexing" phase visible as a progress panel; users can close the tab and return later. `/ask` gates on the index being ready (same 409 contract as `POST /discover`). Re-indexing is user-triggered from Project Settings → Advanced when the warehouse schema changes. Qdrant and an embedding provider are now required. Defaults from the spike against a real 2K-table ERP: Bedrock `qwen.qwen3-32b-v1:0` for blurbs × OpenAI `text-embedding-3-large` for embeddings (perfect MRR, ~$0.60 / 2K tables, ~6 min wall-clock at 8 parallel workers). Reasoning-class models (DeepSeek R1, o1/o3/o4-mini, Claude extended-thinking) are rejected at save time — their `<think>` channel doesn't carry user-visible text through Converse/Chat. New endpoints `GET /api/v1/projects/{id}/schema-index/status`, `POST /schema-index/retry`, `POST /reindex`. Crash-recovery sweep on API boot resets projects stuck in "indexing" > 2h (previous API crash left them orphaned). New `docs/guides/schema-indexing.md` covers the lifecycle, defaults, cost envelope, and re-index triggers.

- **Live debug-log tail on the running discovery panel** — When a discovery is in progress, the project page now renders a per-query tail of every LLM call (`create_message`) and SQL execution (`execute_query`) the agent performs. Rows are expandable — click any entry to reveal the full SQL that ran or the full LLM response (capped at 4 KB server-side, UTF-8-safe). The panel is off by default; toggle it per-project under **Project Settings → Advanced → "Show debug logs during discovery"** (persists in `localStorage`, keyed per project). New endpoint `GET /api/v1/runs/{runId}/debug-logs?since=<RFC3339>&limit=<n>` returns a lean projection — full LLM system prompts and raw query result rows stay in Mongo and never cross the wire. Compound index `(discovery_run_id, created_at)` added so the tail doesn't do an in-memory sort per poll.

- **LLM model catalog + multi-wire cloud dispatch** — New `libs/go-common/llm/modelcatalog` package centralises every cloud-hosted model we ship with its **wire format** (`anthropic`, `openai-compat`, or `google-native`), max output tokens, and list pricing. - **User-count probe routed through the self-healing executor with grounded `FixOpts`** — `services/agent/internal/validation/user_count_validator.go` used to issue three hardcoded `COUNT(DISTINCT user_id) FROM <table>` probes via the raw warehouse client, and silently gave up when none matched the warehouse's column conventions — letting every insight's affected_count claim through unverified on warehouses with non-`user_id` user-id columns (e.g. `KULLANICI_ID`, `customer_id`, `USER_ID_HASH`). The validator now takes an optional `Executor` (the same `SelfHealingExecutor` interface the insight validator uses) and forwards a rendered `FixOpts.VerificationContext` on every probe call. The context is built from the union of all `query_data` step IDs in the exploration log — run-wide evidence, since the user-count probe is single-shot per run rather than per-insight. When the hardcoded `user_id` probe fails with `Invalid column`, the SQL fixer sees the source-step SQL and substitutes the real column name on retry. Without an `Executor` wired, the validator falls through to direct `warehouse.Query` (legacy behaviour preserved for tests and the orchestrator's pre-RunDiscovery construction window). Layer 4 of the verification-grounding fix; closes the loop on `plans/PLAN-INSIGHT-VERIFICATION-GROUNDING.md` §4.4.

- **Verifier owns a `lookup_schema` tool loop for cross-table grounding** — The insight verifier is no longer a single-shot LLM call. With a `SchemaProvider` wired (Layer 3 of the verification-grounding fix), it runs up to `MaxLookupsPerVerification = 6` `lookup_schema` rounds per insight before emitting the final SELECT COUNT — letting the model fetch column detail for tables that `source_steps` did not cover. Lookup results land in the rendered `VerificationContext` after the source-queries block, so the SQL fixer benefits from the same evidence on retry. A shared action parser (`ai.ParseAction(response, allowed)`) takes an allow-list — the explorer's `parseAction` becomes a thin wrapper passing `nil` (any action), the verifier passes `[]string{"lookup_schema", "query_data"}` so the model can't "complete" mid-verify and explorer-only future actions don't silently leak in. When the lookup budget is exhausted before the model emits a query, one forced final round (NOT counted against the budget) is allowed; failure to emit a query after that surfaces as `validation.status="error"` with a clear telemetry signal rather than reusing an arbitrary fallback. Without a `SchemaProvider` wired, the verifier falls through to a single-shot generation (Layer 1 + 2 only). Implementation: new `runVerificationLoop` in `services/agent/internal/validation/insight_validator.go`, new `RenderVerificationContextWithLookups` in `services/agent/internal/validation/render`, `(e *ExplorationEngine).extractJSON` lifted to a free function alongside `ai.ParseAction`. Layer 3 of the verification-grounding fix; Layer 4 (same grounding for `user_count_validator`) follows. Background: `plans/PLAN-INSIGHT-VERIFICATION-GROUNDING.md` §4.3 / §6.6.

- **Self-healing SQL fixer carries per-call verification context** — `SQLFixer.FixSQL` takes a new `queryexec.FixOpts` argument with a `VerificationContext` field. The validator forwards the same source-step grounding evidence the verification prompt was built on, so when the warehouse rejects the verifier's SQL with an `Invalid column name` error the fixer has authoritative names to substitute on retry rather than re-emitting the same hallucination. `QueryExecutor.ExecuteWithFixOpts` is the new entry point; `Execute` is a thin shim that calls it with empty `FixOpts{}` (exploration callers stay zero-opts). Each warehouse `prompts/sql_fix.md` template (BigQuery, Redshift, Snowflake, Postgres, Databricks, MSSQL) gains a conditional `{{#VERIFICATION_CONTEXT}}…{{/VERIFICATION_CONTEXT}}` block stripped from the rendered prompt when the verification context is empty (the explore path) and kept verbatim when populated (the validate path). Layer 2 of the verification-grounding fix; Layers 1 (verifier prompt grounding), 3 (verifier-side `lookup_schema` loop), and 4 (same grounding for `user_count_validator`) ship in adjacent PRs. Background: `plans/PLAN-INSIGHT-VERIFICATION-GROUNDING.md` §4.2.

- **Insight verifier grounded on cited source-step SQL** — The insight verification phase used to receive only the table-level catalog (one line per table, no column names). On warehouses with non-English or abbreviated columns the verifier hallucinated names; a customer report on 2026-04-30 against an MSSQL Netsis-style schema saw 9 of 10 insights end with `validation.status = "error"` and `Invalid column name 'TARIiH' / 'STHAR_SUBE' / 'SUBEKODU' / …`. - **Analysis phase: vector-ranked step selection + per-step compact digest** — The analysis phase no longer ships every keyword-matched exploration step's full `query_result` blob into the LLM prompt. - **`LLM_TIMEOUT` env var now applies to the API process too — single deployment-wide LLM HTTP timeout across the whole stack** — Before this change the agent had `LLM_TIMEOUT` (Go duration) wired through `internal/config` into the provider factory's `cfg["timeout_seconds"]`, but the API process — which builds its own LLM provider for `/ask`, pack-gen, and the enterprise executive-summary plugin — had no env-var path at all. The per-project `timeout_seconds` (set in the dashboard's LLM config) was the only knob, and four providers (Claude, Bedrock, Vertex AI, Azure Foundry) shipped with hardcoded 60s–300s defaults while OpenAI and Ollama outright ignored `timeout_seconds` and pinned 5 minutes in `http.Client{}`. Long-form generations (e.g. executive summaries on Opus-class models with dozens of insights) regularly bumped that 5-minute ceiling and surfaced as `context deadline exceeded` — easy to misread as a token-budget issue. **What changed:** a new `llm.ResolveHTTPTimeout(cfg, fallback)` helper in `libs/go-common/llm/timeout.go` resolves the effective HTTP timeout in one place — `cfg["timeout_seconds"]` (integer seconds, per-project) → `LLM_TIMEOUT` env var (Go duration, deployment-wide) → the provider's historical fallback. All six providers (`claude`, `bedrock`, `vertex-ai`, `azure-foundry`, `openai`, `ollama`) call the helper from their factory; the OpenAI and Ollama constructors gain a `timeout time.Duration` parameter so the factory can plumb the resolved value through (callers passing `0` get the provider default, so tests stay terse). The agent's existing `LLM_TIMEOUT` → `cfg["timeout_seconds"]` pipeline is unchanged — the new env-var fallback only fires when cfg is silent, which is the API process's path. Net effect: setting `LLM_TIMEOUT=15m` once (on the agent **and** the API container) raises every LLM HTTP timeout to 15 minutes; setting per-project `timeout_seconds` in the dashboard still overrides per tenant. Invalid env values (`LLM_TIMEOUT=fifteen-min`, `LLM_TIMEOUT=0s`) fall through to the provider default rather than crashing. Coverage: 17-case table-driven test on `ResolveHTTPTimeout` (cfg wins, env fills in, fallback applies; integer / negative / zero / non-numeric / empty / nil cfg; duration / mixed-unit / negative / zero / bare-int / garbage / whitespace env); a per-provider `*_FactoryWiresTimeout` test on each of the six providers that exercises all three resolution branches through the registered factory (`gollm.NewProvider`) to catch accidental disconnects between `init()` and the helper; a `NewOpenAIProvider`/`NewOllamaProvider` constructor test pinning the "non-positive timeout means default" contract. Documentation: `docs/reference/configuration.md` clarifies that `LLM_TIMEOUT` is read by both the agent and the API and lists the per-provider fallbacks; a new LLM Behavior section under "API Configuration" surfaces the env var on the API side; `helm-charts/decisionbox-api/values.yaml` and `docker-compose.yml` ship a commented-out `LLM_TIMEOUT: "15m"` example on the API service with a note to mirror the same value on the agent.

- **Exploration prompts now render dialect-correct table refs — eliminates SQL-fix retry tax on non-BigQuery warehouses** — Prior to this change, every domain-pack `exploration.md` hardcoded BigQuery-style table refs (`` `dataset.table` `` with backticks) and the orchestrator never injected the warehouse's actual SQL dialect into the prompt. The result: a project connected to MSSQL (or Postgres, Snowflake, Redshift) saw BigQuery examples, generated BigQuery-shaped SQL, paid one full SQL-fix LLM call per query to rewrite each statement into the connected dialect — on a 30-step exploration that's ~30 extra LLM round-trips per run, every run. The validator was only half-fixed: it already injected dialect via `**SQL Dialect**: %s` (`insight_validator.go:552`) but immediately undid that by hardcoding _"ALWAYS use fully qualified table names with backticks: `dataset_name.table_name`"_ in the very same prompt. **What changed:** (1) New `QuoteRef(parts ...string) string` method on the `warehouse.Provider` interface — implemented on all six shipped providers (BigQuery / Databricks emit `` `dataset`.`table` ``, PostgreSQL / Redshift / Snowflake emit `"dataset"."table"`, SQL Server emits `[dataset].[table]`) on top of a colocated `warehouse.QuotePartsWith(open, close, parts)` helper so the filter-and-join logic isn't duplicated. (2) New `substituteDialectTokens` helper in `services/agent/internal/discovery/prompt_subs.go` substitutes two new placeholders in every prompt path (exploration, analysis, recommendations, base context): `{{DIALECT}}` → `provider.SQLDialect()` and `{{REF:tablename}}` → `provider.QuoteRef(firstDataset, "tablename")`. (3) The insight validator's `buildVerificationPrompt` renders example refs through `QuoteRef` and replaces the backtick-only rule with dialect-neutral wording. (4) The `schema_discovery.go` fallback at line 230 (taken when a provider doesn't implement `SampleQueryBuilder` — none today, but the fallback is there for the future) now quotes through `QuoteRef` instead of emitting BigQuery-shaped backticks. (5) Every shipped domain-pack exploration prompt (gaming, social, ecommerce, system-test, and all their category overrides — 12 files) gains a `**SQL Dialect**: {{DIALECT}}` line at the top and replaces 64 `` `{{DATASET}}.tablename` `` occurrences with `{{REF:tablename}}` placeholders; the agent's internal `lookup_schema` array argument continues to take the unquoted `dataset.table` form because that's a tool argument, not warehouse SQL. **Backward-compatible by construction:** the substitution is a no-op on any custom domain pack whose prompts don't contain the new placeholders, so existing deployments keep working unchanged. Customers with custom domain packs upgrade by editing their `exploration.md` (add the `**SQL Dialect**: {{DIALECT}}` line; replace `` `{{DATASET}}.tablename` `` with `{{REF:tablename}}`; soften any "use backticks" rule) — see `docs/reference/prompt-variables.md` for the full migration recipe. Coverage: new `QuotePartsWith` unit tests (13 cases including BigQuery / Postgres / MSSQL delimiter pairs, single / multi / empty / whitespace parts, embedded-dot pass-through); per-provider `QuoteRef` tests on all six providers covering dataset.table, catalog.schema.table, single-part, empty, reserved-word, leading-underscore, and case-preservation cases; `substituteDialectTokens` tests covering both placeholders separately and combined, repeated occurrences, identifier-character constraints (underscore-start ok, leading-digit rejected, dot rejected), nil provider passthrough, empty-refDataset single-part rendering, and malformed-delimiter pass-through. Documentation updated in `docs/concepts/providers.md`, `docs/guides/adding-warehouse-providers.md` (interface signature + method table + Snowflake-example QuoteRef impl), `docs/reference/prompt-variables.md` (full reference for both new placeholders), `docs/guides/customizing-prompts.md` (placeholder table), and `docs/guides/creating-domain-packs.md` (template prompt + Rules section).

- **Schema cache write no longer fails on wide-row warehouse tables** — On data-warehouse-class tables backed by `VARCHAR(MAX)` / `XML` / `JSON` / `VARBINARY` / wide nested-struct columns (Oxxo MSSQL DW, FINPORT-class), a single `SELECT TOP 5 *` sample row routinely exceeds 1 MB; five sample rows × even modest column counts trip Mongo's hard 16 MB BSON document limit on the `project_schema_cache` write. The indexer logged the failure as a warning (`schema_indexer: schema-cache save failed; next run will rediscover`) and continued, leaving `schema_index_status: ready` while `project_schema_cache` was empty for the project — every subsequent discovery run failed with `schema cache lookup failed: schema cache is empty for this project`. `SchemaCacheRepository.Save` now caps cached sample rows at 3 per table and truncates each string / `[]byte` value before insertion: text-shaped values (Go `string` and UTF-8 `[]byte`) are truncated at a **rune boundary** to 256 runes (so multi-byte UTF-8 codepoints are never split — BSON rejects invalid UTF-8) with the marker `…(truncated, original N chars)`; non-UTF-8 `[]byte` (typical for `VARBINARY` / `BLOB` / `FILESTREAM`) is **base64-encoded** at the 256-byte prefix with the marker `…(truncated binary, original N bytes, base64)` so the cache row stays storable as a BSON string without losing fidelity. Non-string values (numbers, bools, time.Time) pass through. New unit tests pin the contract (`schema_cache_repo_test.go`) including a Turkish multi-byte-rune split regression, a non-UTF-8 binary round-trip via base64, and an adversarial 1000-distinct-column × 6-row × oversized-value case that asserts the capped output stays comfortably under the 16 MB cap.

- **Dashboard API proxy no longer caps long-running requests at 5 minutes** — Next.js middleware uses Node's global `fetch` (undici), whose default `headersTimeout` and `bodyTimeout` are both 300_000 ms. That cap killed long synchronous endpoints — most visibly pack-generation synth on Opus, which routinely runs 5–8 minutes against a 270k-char prompt. The dashboard's `middleware.ts` now uses `undici.fetch` with an explicit `Agent` whose timeouts are raised to 20 minutes; the middleware also pins itself to the Node.js runtime so the dispatcher option is honoured. Browser-side `fetch` has no comparable timeout, so raising the proxy is sufficient end-to-end. New jest test (`src/__tests__/proxyTimeout.test.ts`) asserts the dispatcher contract with a slow stub server: a short-Agent rejects with `UND_ERR_HEADERS_TIMEOUT`, the 20-minute Agent waits for a multi-second response, and the timeout constant must stay ≥ 15 minutes. `undici` added as an explicit dashboard dependency. Pack-gen will eventually move to an async-job model that eliminates the synchronous HTTP path entirely; this fix unblocks the synchronous path until then.

- **Bedrock cross-region inference profiles resolve to the correct max-output-tokens cap** — `modelcatalog.Lookup` now strips a leading geo qualifier (`us.`, `eu.`, `apac.`, `jp.`, `au.`, `global.`) from a Bedrock model ID and retries the catalog lookup against the bare ID. Without this fallback, an `us.anthropic.claude-opus-4-7-v1:0` lookup missed the catalog and `llm.GetMaxOutputTokens` fell through to the provider's `_default` of 16384 — small enough that an Opus 4.7 pack-generation synth (full DomainPack JSON, 271k-char prompt) truncated mid-output, retried, and ran past the dashboard's 5-minute proxy budget. Bedrock now correctly resolves the 128k Opus cap (and 64k Sonnet/Haiku caps, 32k legacy Opus 4 cap) for every cross-region inference profile alias of an already-catalogued model. The strip is Bedrock-specific so Vertex / Azure model IDs that happen to start with `us.` or `global.` are unaffected.

- **Schema discovery no longer costs one LLM call per table on non-BigQuery warehouses** — `SchemaDiscovery.getSampleData` used to emit a hardcoded BigQuery/MySQL-style query (`` SELECT * FROM `dataset.table` … LIMIT 5 ``) for every warehouse provider, which fails on T-SQL (MSSQL), PostgreSQL/Redshift/Snowflake (double-quoted identifiers), and any dialect without `LIMIT`. The SQL fixer would rewrite it per table — an extra Claude round-trip (~5s + ~8 KB input tokens) for every table in the warehouse before the first sample could be fetched. New optional `warehouse.SampleQueryBuilder` interface in `libs/go-common/warehouse` lets a provider supply its own dialect-native sample query. All six community providers implement it: BigQuery (`` `dataset.table` `` + `LIMIT`), MSSQL (`SELECT TOP 5 * FROM [schema].[table]`), PostgreSQL and Redshift (`"schema"."table"` + `LIMIT`), Snowflake (`"SCHEMA"."TABLE"` + `LIMIT`, preserving case), Databricks (`` `schema`.`table` `` + `LIMIT`). Providers that don't implement it still fall back to the BigQuery-style legacy — behaviour for any third-party provider is unchanged. Unit tests per provider + schema-discovery routing test covering both the builder and fallback paths.

- **Exploration terminated after 2–18 steps on reasoning-model LLMs** — The exploration response parser treated any response it couldn't cleanly decode as a "complete" signal: JSON without `query`/`done`/`action` was silently assumed done, and responses with no JSON fell through to a plain-text substring match on `"done"` / `"complete"` / `"finished"` — so a Qwen3 or DeepSeek-R1 thinking block mentioning *"I'm done analyzing area 1"* or *"the query completed"* ended the run. The parser is now strict: it walks every balanced JSON object (string-literal-aware, so `}` inside SQL strings no longer breaks the count), prefers the last block with a known action key (reasoning preambles no longer hijack the parse), and rejects anything without one. Unparseable responses are re-prompted up to three times with a reformat nudge instead of silently terminating. A new `MinSteps` exploration option (surfaced as `--min-steps` on the agent binary) rejects premature `done` signals, injects a nudge with the current/required step count, and continues — guarding against models biased toward early termination even with valid JSON. Covered by 40+ new unit tests including a Qwen3-style regression reproducing the original failure.

- **Ollama output tokens cap too low on large modern models** — The Ollama provider previously set a single `MaxOutputTokens["_default"] = 8192` for every model, which truncated long analyses on large reasoning and 2025/2026-generation models. Raised the default floor to 16384 and added documented per-model caps for the biggest Qwen, Gemma, DeepSeek, and Meta (Llama) families actually available in the Ollama library: 65536 for Qwen 3.5 / 3.6 (`qwen3.6:35b-a3b` tops out at 81920 per the model card), 32768 for DeepSeek R1 reasoning variants and Qwen 3 (32B / 235B), 16384 for Qwen 2.5 72B, DeepSeek V3, and Gemma 3 27B, and 8192 for Llama 4 Scout / Maverick, Llama 3.x (including the `llama3.1:8b` tag we recommend in our quick-start docs), Llama 3, and Gemma 2 (2B / 9B / 27B). Bare model names (`qwen3`) and `:latest` aliases are both mapped, and every documented Ollama size tag in the capped families is listed explicitly so the exact-match `GetMaxOutputTokens` lookup never silently falls through to the higher default.

- **Docker Compose Quick Start: `decisionbox-agent` missing from API image** — The API container's default `RUNNER_MODE=subprocess` spawns the agent via `exec.Command("decisionbox-agent", ...)`, but `services/api/Dockerfile` only shipped the `decisionbox-api` binary. Starting a discovery failed with `exec: "decisionbox-agent": executable file not found in $PATH`. The API image now also builds and installs `decisionbox-agent` into `/usr/local/bin`, so `docker compose up -d` works end-to-end out of the box. Kubernetes deployments are unaffected — they use `RUNNER_MODE=kubernetes` and run the agent as a Job from its own image.

### Changed

- **`apiserver.Run()` now owns subcommand routing** — The `backfill-embeddings` subcommand is dispatched inside `apiserver.Run()` instead of from `services/api/main.go`. Custom API binaries (e.g., enterprise) that call `apiserver.Run()` automatically get all subcommands wired up, closing a gap where subcommands were unreachable from non-community entry points.

### Fixed

- **Go dependency security upgrades** — Resolved 18 Dependabot alerts (1 critical, 2 high, 15 medium) across 10 Go modules: `google.golang.org/grpc` (authorization bypass), `github.com/go-jose/go-jose/v3` (JWE decryption panic), `golang.org/x/oauth2` (input validation), and the AWS SDK v2 `bedrockruntime` / `eventstream` / `s3` packages (EventStream DoS). Dashboard Dockerfile now runs `apk upgrade --no-cache` to pick up Alpine security patches on rebuild.
- **IP allowlist wiring on AWS and GCP** — `setup.sh` was creating IP restriction resources in Terraform but never passing them to Helm, leaving the ALB / GCE ingress open to `0.0.0.0/0`. AWS now uses the `alb.ingress.kubernetes.io/inbound-cidrs` annotation (keeping the controller's backend SG management intact), and GCP now creates a `BackendConfig` CRD and annotates the dashboard Service with `cloud.google.com/backend-config` to attach the Cloud Armor policy.
- **Orphaned AWS `ip_allowlist` security group removed** — `aws_security_group.ip_allowlist` and its three supporting rules were no longer attached to anything after the switch to `inbound-cidrs`. Removed from the `terraform/aws` module along with the proxied output.
- **Terraform LLM IAM now granted to the API role, not just the agent** — `enable_bedrock_iam` / `enable_vertex_ai_iam` on AWS and GCP now also attach to the API's IRSA / Workload Identity SA. Previously the `/ask` endpoint (and any future API-side LLM call) returned 500 on EKS/GKE deployments with LLM IAM enabled because the API role had no `bedrock:*` or `aiplatform.user` permissions.
- **K8s test connection response parsing** — `extractJSONObject` now scans pod logs from the end and skips structured log lines (identified by the `"severity"` key). K8s pods mix stdout (result) with stderr (log lines), so the previous implementation picked up the first JSON line — a structured log, not the agent result — and the dashboard showed "Unknown error". Also adds `pods/log` to the `agent-job-manager` RBAC Role so the API can read agent pod logs.
- **Redshift SQL fix prompt** — The Redshift warehouse provider now returns a Redshift-specific self-healing prompt from `SQLFixPrompt()` (previously returned an empty string, so the agent's self-heal loop had no guidance for Redshift failures). Prompt covers the PostgreSQL features Redshift does not support (`DISTINCT ON`, `FILTER (WHERE ...)`, `LATERAL`, `generate_series`, `string_agg`, `array_agg`, `regexp_matches`, `FORMAT`), Redshift-native alternatives (`QUALIFY`, `LISTAGG`, `SUPER` + `json_extract_path_text`, `DATEADD`/`DATEDIFF`/`GETDATE`, `CONVERT_TIMEZONE`), and 17 common Redshift error patterns. `make lint-go` now includes the Redshift provider and CI has a matching `Lint Redshift warehouse provider` step.

### Removed

- Removed all Go code from `domain-packs/*/go/` directories (dynamic domain packs).
- Removed `libs/go-common/domainpack` package (registry, interfaces).
- Removed domain pack blank imports from agent and API server.
